# 环境形式化说明

## 1. 文件与目标

环境形式化代码位于：

```text
docs/Mathematical_logic/examples/NesyLinkEnvironment.lean
```

该文件建立五个 Task 共用的符号环境，不包含固定地图、固定宝箱坐标或固定
行动路线。后续每个 Task 只需要提供具体房间、对象初态和目标，再复用这里的状态类型、转移关系与定理。

形式化对象是 **CNN/像素识别之后的 tile 级符号层**。Python Agent 从图像得到玩家、墙、宝箱、怪物等符号；Lean 证明 planner 和 safety shield 在这些符号输入正确时遵守环境规则。CNN 对所有图像都正确 不属于本课程要求的证明范围。

## 2. 总体结构

### 2.1 基础类型

对应 Lean 文件“一、基础标识、坐标、方向和动作”。

- `Position`：整数坐标。使用整数是为了让向左或向上越界产生负坐标，再由
  `inBounds` 拒绝，而不是被自然数减法截断成 0。
- `Bounds`：房间宽、高以及二者为正的证明。
- `Direction`：north、south、west、east。
- `Action`：wait、up、down、left、right、slotA、slotB，与接口七种动作对应。
- `ObjectId`、`RoomId`：对象和房间的稳定标识。

**对应评分点：状态、动作定义是否齐全。**

### 2.2 对象和属性

对应 Lean 文件“二、对象的静态类型与运行时属性”。

- `Chest`：ID、位置、loot、可见性、开启状态，以及
  `ChestRevealCondition` 隐藏宝箱揭示条件。
- `Monster`：ID、位置、怪物类型、HP、接触伤害。
- `Trap`：spike/abyss、伤害、重生点、激活状态、是否一次性。
- `Button`：踩踏按钮及 pressed 状态。
- `Switch`：相邻交互开关和目标桥 ID。
- `Bridge`：横向、纵向 tile 集合及当前朝向。
- `Npc`：ID、位置和对话文本。
- `Inventory`：钥匙、金币、物品。
- `PlayerState`：位置、朝向、HP、最大 HP、背包、临时举盾状态。
- `Loot`：钥匙、金币、治疗、物品，不把宝箱奖励固定成某一种。

`RoomState` 保存一个房间的全部对象；`WorldState` 使用
`RoomId → RoomState` 保存所有房间，并用有限 `roomIds` 标识 dungeon
模板中的实际房间，因此跨房间后原房间的变化仍然存在。

**对应评分点：对象、属性、状态定义是否齐全；是否支持多房间关键机制。**

### 2.3 出口和条件

`Requirement` 覆盖 Python schema 中的主要条件：

| Lean 条件 | Python 含义 |
|---|---|
| `free` | normal exit，无额外要求 |
| `keys count consume` | `key_count` 与 `consume_key` |
| `buttonPressed id` | `button_pressed` |
| `ownsItem item` | `item` |
| `allMonstersDefeated` | `all_monsters_defeated` |
| `both left right` | 同时满足多个条件 |

`Exit` 还记录出口位置、方向、目标房间、目标 spawn、是否完成任务以及
`opened` 持久状态。`exitRequirementSatisfied` 允许已经打开的锁门再次通过；
`spendExitRequirement` 只在锁门第一次打开且配置要求时消耗钥匙。

**对应评分点：能够刻画钥匙门、条件门、清怪门和多房间切换。**

## 3. 可通行性与安全性

### 3.1 物理可通行

```lean
def canEnter (r : RoomState) (p : Position) : Prop
```

目标位置必须：

1. 位于房间边界内；
2. 不是墙、NPC 或可见宝箱；
3. 不是没有活动桥覆盖的 gap。

打开后的宝箱仍然是可见实体，因此依然阻挡。这与项目中已经遇到的“开箱后
错误穿箱并卡住”问题直接对应。定理
`opened_chest_is_still_a_static_blocker` 明确证明了这一点。

### 3.2 Agent 安全 tile

```lean
def safeTile (r : RoomState) (p : Position) : Prop
```

`safeTile` 在 `canEnter` 基础上额外排除活动陷阱和怪物。这里刻意区分：

- 陷阱在物理上允许进入，所以它不能被错误定义为墙；
- Agent 的 safety shield 可以主动选择不进入陷阱。

**对应评分点：移动语义合理；能够区分阻挡物和危险物；支持安全规划。**

## 4. 单步转移语义

核心关系为：

```lean
Step s action t events
```

它表示状态 `s` 经一个符号微步可以到达 `t`，并产生事件列表 `events`。
Python 一次 tick 内依次执行玩家动作、出口判定、tile 效果、怪物更新和接触
结算；Lean 将这些阶段拆成可组合微步。玩家微步使用真实动作，自主怪物阶段
使用 `wait` 表示没有新的玩家输入。

### 4.1 移动与碰撞

- `movePlain`：移动到普通可通行 tile，并更新位置和朝向。
- `moveButton`：移动后自动把按钮设为 pressed。
- `moveTrapSurvive`：进入陷阱后仍存活，扣血并回到合法重生点。
- `moveTrapFatal`：扣血后 HP 为 0，不执行重生，停留在触发 tile。
- `moveBlocked`：越界或遇到墙、宝箱、NPC、gap 时位置不变，但朝向更新。

这把 Python 的若干 pixel action 抽象为一次完整 tile action。形式化不描述
人物位于两个 tile 之间的动画帧，而描述 tile 动作结束后的稳定符号状态。

### 4.2 宝箱和资源

- `openChest` 要求宝箱存在、可见、未开启且与玩家相邻。
- 开启后根据 `Loot` 分别增加钥匙、金币、物品或治疗。
- 治疗使用 `min maxHp`，保证生命不超过上限。
- slot A 按 Python 顺序处理：可开宝箱优先，其次 NPC、switch，最后才允许
  使用剑攻击；后续分支都带有排除更高优先级交互的前提。

### 4.3 攻击、怪物和盾牌

- `attackDamage`：持有剑、怪物位于朝向前方一格且未被击杀，按
  `swordDamage = 1` 扣除 HP。
- `attackKill`：伤害足以击杀时删除怪物，并按 `monsterKillGold = 1`
  发放击杀金币；规则构造器不能任意选择伤害或奖励。
- `resolveMonsterKill`：若本次击杀清空当前房间，立即执行与 Python 相同的
  后置结算：持久打开包含 `allMonstersDefeated` 条件的门，再遍历所有房间，
  按 `revealOn` 条件揭示隐藏宝箱。
- `monsterKillEvents`：按“击杀、门开启、宝箱揭示”的顺序产生对应事件。
- `monsterMove`：怪物只能走到相邻、界内、非阻挡且没有其他怪物的位置。
- `monsterContact`：未举盾接触怪物时扣血。
- `shieldContact`：举盾时阻止该次伤害，并结束临时举盾状态。
- `shieldUnavailable`：没有盾时使用 slot B 不会凭空获得防御能力。

具体 chaser、patroller、ambusher 的选路和随机数被抽象成非确定选择；但任何
被允许的怪物移动都必须满足共同的地图约束。

### 4.4 按钮、开关和桥

- button 在 `moveButton` 中通过踩踏触发。
- switch 在 `activateSwitch` 中要求与玩家相邻并使用 slot A。
- switch 在当前房间记录 pressed，并可通过 `targetRoom` 旋转其他房间的桥。
- `gapAt` 根据桥当前覆盖的 tile 判断原 gap 是否已经可通行。

### 4.5 出口和多房间

- `useExit` 要求玩家位于出口、朝向出口、满足 `Requirement`，并继续执行
  与出口方向一致的移动动作，而不是使用 WAIT 出门。
- 成功后切换 `currentRoom`，把玩家放到 `targetSpawn`。未开启的 locked exit
  会写回 `opened = true` 并按配置消耗钥匙；以后再次通过不重复扣钥匙。
- `doorOpened` 只在锁门第一次打开时产生，普通出口和已开启锁门不伪造该事件。
- 完成型出口会设置 `completed = true`。
- `exitBlocked` 表示条件不满足，世界状态保持不变。
- Task5 使用 `completeAllChests`：世界中至少存在一个宝箱，且有限房间集的
  所有宝箱都已可见并打开后，才完成世界；零宝箱世界不会真空完成。

**对应评分点：移动、撞墙、陷阱、开箱、攻击、按钮切换、桥、出口等动作
转移语义是否合理；是否刻画关卡关键机制。**

### 4.6 Tick 调度和终止状态

- `PlayerStep` 只允许在 `Running` 状态执行玩家动作。
- `AutonomousStep` 表示怪物移动、接触伤害、盾牌格挡和完成结算。
- `EngineTick` 明确规定先执行一次玩家阶段，再执行零次或多次自主阶段。
- `EngineExec` 是完整 tick 的多步闭包，动作列表只记录玩家输入，自主微步
  保留在对应 tick 内部。
- `dead_state_has_no_player_step` 和 `completed_state_has_no_player_step` 证明
  死亡或已完成的世界不能继续接收玩家动作。

## 5. 目标谓词

原子目标包括：

- `alive`
- `HasKey`
- `HasItem`
- `ChestOpened`
- `AllMonstersDefeated`
- `ButtonPressed`
- `RoomReached`
- `ExitReached`
- `WorldCompleted`

`Goal` 将它们包装为统一目标语言，`Goal.both` 可以递归组合多个目标，
`GoalHolds` 给出目标在世界状态上的逻辑解释。

示例：

```lean
Goal.both .hasKey (.hasItem .sword)
Goal.both .monstersDefeated .worldCompleted
```

后续五关可以分别定义 `Task1Goal` 至 `Task5Goal`，而无需修改通用环境。

**对应评分点：目标谓词定义是否齐全、能否表达五关任务链。**

## 6. 多步轨迹

`Exec s actions t` 是 `Step` 的多步闭包：

- `Exec.nil`：空动作序列保持状态不变；
- `Exec.cons`：执行一个合法 `Step`，再执行剩余动作。

它是环境形式化与策略证明之间的接口。后续可证明：

```text
策略生成 actions
→ Exec initial actions final
→ GoalHolds final taskGoal
```

## 7. 已证明的基础性质

### 7.1 房间状态

- `setRoom_same`：更新目标房间后读取到新房间。
- `setRoom_other`：更新一个房间不会改变其他房间。

### 7.2 生命与资源

- `damage_hp_le`：伤害不会增加 HP。
- `damage_preserves_hp_bound`：伤害保持 `hp ≤ maxHp`。
- `collectLoot_hp_bound`：任何 loot 都保持生命上界。
- `heal_never_exceeds_max_hp`：治疗不会超过最大生命。
- `key_loot_increases_keys`：钥匙奖励按指定数量增加。

### 7.3 移动安全

- `canEnter_inBounds`：可进入位置一定在边界内。
- `canEnter_not_wall`：可进入位置不是墙。
- `canEnter_not_visible_chest`：可进入位置不是可见宝箱。
- `blocked_move_keeps_position`：被阻挡的移动不改变位置。
- `legal_move_in_bounds`：合法移动不会越界。
- `legal_move_not_into_wall`：合法移动不会进入墙。
- `legal_move_not_into_visible_chest`：合法移动不会进入宝箱。
- `safeTile_is_enterable`：安全 tile 必然物理可通行。
- `step_preserves_validState`：任意合法微步保持玩家界内以及 `hp ≤ maxHp`。
- `engineTick_preserves_validState`：完整 tick 也保持上述不变量。
- `engineExec_preserves_validState`：任意有限完整 tick 轨迹保持上述不变量。
- `engineExec_has_microstep_trace`：完整 tick 轨迹可以展开为合法 `Exec` 微步
  轨迹，连接环境调度语义与已有任务组合证明。

### 7.4 世界与关卡配置良构性

- `WellFormedWorld`：房间 ID 集非空且无重复、当前房间存在、玩家状态合法。
- `RoomConfigurationWellFormed`：检查墙、NPC、宝箱、怪物、陷阱、机关、桥、
  动态 tile 和出口位置均在本房间边界内。
- `ExitTargetsKnown`：每个出口的目标房间属于有限房间集合。
- `ExitSpawnsInBounds`：每个出口 spawn 位于对应目标房间边界内。
- `step_preserves_roomIds`：转移不会偷偷增删房间索引。
- `step_preserves_wellFormedWorld`：在出口目标已知的地图配置前提下，任意
  合法 `Step` 保持核心世界良构性。

`RoomConfigurationWellFormed`、`ExitTargetsKnown` 和 `ExitSpawnsInBounds`
是关卡模板应提供的静态合法性证书；它们不是从任意坏地图自动推导出的结论。

### 7.5 关键机制

- `opened_chest_is_still_a_static_blocker`：开启后的宝箱仍阻挡。
- `opening_chest_marks_open`：开箱后 opened 为 true。
- `damaging_attack_reduces_monster_hp`：有效攻击严格降低怪物 HP。
- `pressing_button_is_monotone`：按钮触发后为 pressed。
- `rotating_bridge_twice_restores_orientation`：桥旋转两次恢复原朝向。
- `shield_contact_preserves_hp`：盾牌成功格挡时 HP 不变。
- `trap_damage_never_increases_hp`：陷阱伤害不会增加 HP。
- `trap_respawn_in_bounds`：合法陷阱重生点位于房间内。
- `fatal_trap_keeps_zero_hp_at_trigger`：致死陷阱不重生，HP 为 0 且停在触发格。
- `matching_hidden_chest_is_revealed`：匹配清怪触发条件的隐藏宝箱变为可见，
  且保持对象 ID 和坐标。
- `clearing_requirement_exit_is_opened`：包含清怪条件的门持久变为 opened，
  且保持出口 ID 和坐标。
- `revealEligibleChests_preserves_chest_positions`：揭示操作不移动任何宝箱。
- `unlockAllMonstersDefeatedExits_preserves_exit_positions`：自动开门不移动出口。
- `resolveMonsterKill_preserves_validState`：完整清怪结算保持玩家界内和 HP 上界。
- `monster_kill_without_room_clear_has_no_clear_events`：尚有怪物时不会错误产生
  自动开门或宝箱揭示事件。

### 7.6 出口与轨迹

- `key_requirement_needs_enough_keys`：钥匙条件成立意味着钥匙足够。
- `consuming_key_requirement_spends_keys`：消耗型钥匙门准确扣除钥匙。
- `unopened_locked_key_exit_spends_exactly`：锁门首次开启准确扣除配置数量。
- `opened_locked_exit_does_not_spend_again`：已开启锁门不会重复扣钥匙。
- `successful_exit_enters_target_room`：成功出口进入目标房间和 spawn。
- `successful_exit_spawn_in_bounds`：合法出口 spawn 位于目标房间内。
- `completed_exit_sets_world_completed`：完成型出口设置世界完成。
- `exec_append`：两段轨迹可以拼接。
- `exec_nil_iff`：空轨迹的终态等于初态。

**对应评分点：证明基本安全性和不变量，特别是“不越界、不进墙、不穿宝箱”。**

## 9. 明确的抽象边界

为了保证报告描述与代码一致，需要说明以下抽象：

1. Python 的 16 次左右 pixel action 被抽象为一次稳定的 tile action。
2. CNN/像素分类的正确性不在 Lean 中证明；Lean 从符号状态开始验证。
3. 怪物具体 AI 和 RNG 不逐行复刻，而用受地图约束的非确定移动表示。
4. 动画中人物跨两个 tile、攻击击退和短暂 stun 没有进入 tile 级状态。
5. `Event` 是形式化转移的输出标签，不是 Agent 偷读的 Python `info`。
6. 具体关卡的初始地图和 Task 目标应在后续 Task 章节中实例化。
7. `Step` 是 Python tick 内部阶段的微步关系，不把自主怪物更新伪装成新的
   玩家动作。

这些简化不会隐藏 planner 的关键义务：任何被 planner 选择的移动仍必须通过
边界、墙、宝箱和 gap 检查；陷阱、怪物、钥匙、按钮、桥和出口条件也都有
明确的符号语义。
