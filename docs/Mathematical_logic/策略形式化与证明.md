# 策略形式化与证明

## 1. 文档目标

策略形式化代码位于：

```text
docs/Mathematical_logic/examples/NesyLinkEnvironment.lean
```

该 Lean 文件前半部分定义通用环境，后半部分依次形式化 Task1、Task2 和
Task5 的策略。本文解释：

1. Python Agent 的哪些部分进入了 Lean；
2. 安全性、可达性、关键子任务正确性和完备性分别如何证明；
3. Task1、Task2、Task5 的证明为什么采用不同强度的前提；
4. 每部分分别对应课程评分中的哪些踩分点。

## 2. 证明边界

### 2.1 Lean 证明覆盖的层

整体流程为：

```text
RGB 图像
   ↓ CNN / 像素分类
符号观测
   ↓ 记忆更新
FSM / 高层目标选择
   ↓
BFS tile 路径
   ↓
可中断动作队列
   ↓
action mask / safety shield
   ↓
环境 Step / Exec
   ↓
Task 目标谓词
```

Lean 重点验证从“符号观测”开始的部分：

- 历史静态阻挡记忆；
- BFS 路径规格；
- FSM 阶段推进；
- 高层目标选择；
- action mask / safety shield；
- 宝箱、攻击、按钮、出口等局部交互；
- 多步轨迹和最终任务目标。

### 2.2 Lean 不证明的层

以下部分被明确放在证明边界之外：

- CNN 对所有渲染图像都分类正确；
- 像素动画的每一个中间帧；
- 玩家位于两个 tile 之间时的视觉框变化；
- 怪物 AI 随机数的具体输出序列；
- Python `deque`、`set`、`dict` 的语言实现正确性。

因此证明通常采用如下形式：

```lean
符号观测/记忆满足正确性契约
→ planner 和 safety shield 输出合法
→ 环境轨迹满足安全性和任务目标
```

这符合课程说明中“机器学习模型本身不要求在 Lean 中全部证明，但需要证明
可验证符号层”的要求。

## 3. 通用证明架构

### 3.1 单步与多步执行

环境使用：

```lean
Step s action t events
```

表示状态 `s` 执行动作 `action` 后，可以到达 `t` 并产生 `events`。

更准确地说，`Step` 是 Python 一次 tick 内部阶段的符号微步：玩家动作保留
真实 action，出口、tile 效果、怪物移动和接触结算可以依次组合；自主怪物
阶段使用 `wait` 标签表示没有新增玩家输入。出口本身使用对应方向动作推进。

`PlayerStep`、`AutonomousStep` 和 `EngineTick` 进一步约束真实调度顺序：
每个 tick 先执行一次仍存活玩家的动作，再执行环境自主微步；死亡或完成后
不能继续开始玩家动作。

加强版动态任务证明使用 `EngineExec` 作为完整 tick 的多步闭包。其动作列表
只记录玩家输入；`engineExec_has_microstep_trace` 证明它可展开为合法 `Exec`
微步轨迹，`engineExec_preserves_validState` 证明整段执行保持状态有效。

策略执行使用：

```lean
Exec initial actions final
```

它是 `Step` 的多步闭包。策略正确性最终都归结为：

```lean
∃ actions final,
  Exec initial actions final ∧
  TaskGoal final
```

`exec_append` 证明两段合法轨迹可以拼接，是三个 Task 组合子任务的核心引理。

### 3.2 路径与 BFS

Task1 中定义：

```lean
TilePath room start route goal
```

其每一步必须：

- 与上一位置四邻接；
- 等于某个方向的 `advance`；
- 满足 `safeTile`。

所以 `TilePath` 不只是位置列表，而是携带完整安全证明的路径证书。

相关性质：

- `tilePath_first_step_safe`：第一步是安全 tile；
- `tilePath_first_step_adjacent`：第一步与起点相邻；
- `bfs_result_is_sound`：BFS 结果确实到达目标集合；
- `bfs_first_move_is_safe`：BFS 输出的第一步安全；
- `tilePath_has_executable_plan`：符号路径能够转换为真实 `Exec` 动作轨迹。

### 3.3 BFS 完备性的准确含义

定义：

```lean
BoundedTileReachable room start depth goal
BfsFrontierComplete room start depth frontier
```

`BfsFrontierComplete` 表示：深度为 `depth` 的 frontier 已覆盖所有长度不超过
该深度的可达位置。

定理：

```lean
bfs_complete_from_frontier_invariant
```

证明只要：

1. frontier 满足按层覆盖不变量；
2. 某目标在给定深度内可达；

则 frontier 一定包含目标。

这是 BFS 完备性的逻辑核心。Lean 当前没有逐行复刻 Python `deque` 和
parent 字典，但证明了 Python BFS 应维护的关键覆盖不变量及其完备性后果。

### 3.4 safety shield

每个 Task 都有对应动作过滤器：

- `task1Shield`
- `task2Shield`
- `task5Shield`

共同逻辑是：

```text
非移动动作：交给对应交互规则
移动动作：
  若满足当前任务的 MoveAllowed → 放行
  否则 → WAIT
```

证明包括：

- 被放行的移动一定物理可进入；
- 不合法移动会被替换为 WAIT；
- 历史反馈确认会卡住的边不会再次放行；
- 普通模式不会主动进入怪物邻域。

这直接对应机器学习策略的“可验证 action mask / safety shield”评分点。

## 4. Task1 策略证明

### 4.1 Python 策略结构

Task1 是静态单房间：

```text
toChest → toExit → done
```

策略先从视觉中发现宝箱，BFS 到宝箱相邻位置，使用 slot A 开箱；确认钥匙后
BFS 到锁门，再向出口方向推进。

### 4.2 静态阻挡记忆

`Task1Memory` 保存历史墙和宝箱位置。

关键定理：

- `remembered_blocker_is_retained`
- `newly_remembered_blocker_is_present`
- `rememberBlocker_preserves_soundness`

这些定理证明：

- 旧阻挡不会因新观测丢失；
- 新观测阻挡会被写入；
- 若写入位置确实是静态阻挡，则更新后记忆仍然可靠。

这解决了“宝箱打开后外观变化，被视觉重新分类为 floor，BFS 错误穿箱”的问题。

### 4.3 FSM 正确性

Task1 阶段定义为：

```lean
Task1Phase.toChest
Task1Phase.toExit
Task1Phase.done
```

每个阶段有自然数 rank：

```text
toChest = 0
toExit  = 1
done    = 2
```

定理：

- `task1_phase_never_regresses`
- `task1_key_advances_to_exit`
- `task1_completion_advances_to_done`

说明 FSM 不会从后续阶段退回前一阶段；拿到钥匙后进入出口阶段；完成后进入
done。

### 4.4 开箱与通关组合

`task1_open_key_chest_gives_key` 证明钥匙宝箱开启后 `HasKey` 成立。

主定理：

```lean
task1_completable_if_subplans_exist
```

其逻辑为：

```text
Exec initial toChest nearChest
→ Step.openChest
→ Exec afterChest toExit atExit
→ Step.useExit
→ WorldCompleted
```

它把两段 BFS 子计划与开箱、出口交互拼成一条完整 `Exec` 轨迹。

### 4.5 公开地图实例

公开 Task1 地图的坐标只在离线实例证明中出现。

已证明：

- `task1_public_bfs_path_to_chest`
- `task1_public_bfs_path_to_exit`
- `task1_public_chest_is_adjacent`
- `task1_public_chest_phase_reachable`
- `task1_public_exit_phase_reachable`

两条路径的每个 tile 都显式经过 `safeTile` 检查，因此不是仅给出一条未经
验证的固定路线。

### 4.6 Task1 证明强度

Task1 地图静态、没有移动怪物，因此可以得到较强结论：

```text
宝箱可达 ∧ 出口可达 ∧ BFS frontier 完整
→ FSM 可以完成 Task1
```

**对应踩分点：**

- FSM/planner 形式化；
- BFS 健全性与有界完备性；
- action mask 安全性；
- 开箱、钥匙门正确性；
- 具体关卡可达性；
- 成功轨迹满足最终目标。

## 5. Task2 策略证明

### 5.1 Python 策略结构

Task2 有动态 chaser：

```text
toMonster → toChest → toExit → done
```

与 Task1 的主要区别：

- 怪物会移动；
- 移动队列每帧都可以被打断；
- 战斗需要安全攻击站位、朝向和多次挥剑；
- 非战斗阶段需要避开怪物邻域。

### 5.2 三帧清怪确认

Python Agent 不会因单帧漏检立刻切换阶段。

Lean 中定义：

```lean
updateMissingFrames
monsterCleared
```

并证明：

- `three_missing_frames_confirm_clear`
- `visible_monster_resets_missing_frames`

即连续三帧没有怪物才确认清怪，一旦再次看见怪物计数归零。

### 5.3 Task2 FSM

关键定理：

- `task2_phase_never_regresses`
- `task2_clear_advances_to_chest`
- `task2_key_advances_to_exit`
- `task2_nonempty_queue_delays_phase_change`

最后一个定理说明：旧动作队列没有清空时，不立即切换目标，避免阶段切换后继续
执行上一阶段的像素动作。

### 5.4 战斗形式化

定义：

```lean
FacingMonster
AttackPosition
AttackReady
```

其中：

- `AttackPosition` 要求玩家位于安全 tile 且与怪物相邻；
- `FacingMonster` 要求怪物位于玩家前方一格；
- `AttackReady` 还要求玩家持剑、怪物存在且 HP 为正。

玩家朝向不正确时，Python 会向怪物方向发出很短的像素动作。Lean 增加
`Step.faceMonster`：

- 仅当目标格确实有怪物时允许；
- 只更新朝向；
- 不把玩家完整移动到怪物格。

相关定理：

- `face_monster_keeps_player_position`
- `face_monster_step_is_position_safe`

### 5.5 怪物 HP 与有限击杀

剑攻击力和击杀金币分别由固定常量 `swordDamage = 1` 与
`monsterKillGold = 1` 给出，证明规则不能自行选择更高伤害或奖励。

最后一只怪物死亡时，`resolveMonsterKill` 还会执行 Python 引擎的清怪后置
结算：自动打开清怪条件门，并根据 `ChestRevealCondition` 跨房间揭示隐藏
宝箱。因此 Task2/Task5 后续规划所使用的可见宝箱和门状态来自形式化转移，
不再作为未经解释的外部状态跳变。

定义：

```lean
hpAfterSwordHits hp hits = hp - hits
```

证明：

- `one_sword_hit_strictly_decreases`
- `hp_sword_hits_are_sufficient`
- `task2_attack_damage_strict_progress`
- `task2_attack_kill_removes_target`

因此正 HP 每次有效攻击都严格下降，初始 HP 为 `hp` 时至多 `hp` 次有效命中
即可归零。

### 5.6 动态安全层

定义：

```lean
OutsideMonsterDanger
Task2MoveAllowed
QueueMustInterrupt
```

战斗阶段可以到怪物相邻攻击位；开箱和出口阶段必须远离怪物 tile 及其一格
邻域。

证明：

- `task2_allowed_move_is_safe`
- `task2_noncombat_move_avoids_monster_neighborhood`
- `task2Shield_blocks_disallowed_move`
- `task2Shield_allowed_move_is_safe`
- `unsafe_queued_move_must_interrupt`
- `interrupted_move_is_masked_to_wait`

这对应 Python“每帧重新视觉检测、队列可中断、安全层重新过滤”的机制。

### 5.7 Task2 组合正确性

主定理：

```lean
task2_completable_if_subplans_exist
```

加强定理 `task2_engine_execution_is_safe_and_complete` 进一步要求动态轨迹由
`EngineExec` 给出，并同时证明终态完成、玩家存活以及 `ValidState`，避免用
任意顺序的怪物微步冒充合法 tick。

组合逻辑为：

```text
BFS 到怪物
→ 战斗 Exec 清空怪物
→ BFS 到宝箱
→ 开箱获得钥匙
→ BFS 到条件出口
→ 满足清怪与钥匙条件
→ WorldCompleted
```

### 5.8 动态环境下的条件完备性

不能诚实地证明“任意怪物随机轨迹下策略都一定终止”。例如怪物理论上可能长期
占据唯一通路。

因此定义：

```lean
EventuallyCombatClears
```

它表示动态怪物最终进入可攻击状态，并存在有限合法战斗轨迹将其清空。

Task2 完备性准确表述为：

```text
目标可达
∧ 怪物最终可攻击
∧ 有限攻击可清怪
∧ 玩家能够存活
→ Task2 可完成
```

### 5.9 公开地图实例

已证明：

- 初始位置到安全攻击位的 BFS 路径；
- 攻击位与怪物相邻；
- 攻击位本身安全；
- HP=2 的怪物需要两次剑击；
- 第一剑执行 `attackDamage`，HP 从 2 降到 1；
- 第二剑执行 `attackKill`，怪物从列表删除；
- `Exec [.slotA, .slotA]` 清空怪物；
- 清怪后到宝箱的路径；
- 开箱后到 west 条件出口的路径。

主要定理包括：

- `task2_public_combat_exec`
- `task2_public_combat_clears_monster`
- `task2_public_all_navigation_phases_reachable`

**对应踩分点：**

- 动态环境策略形式化；
- 可中断动作队列；
- safety shield；
- 朝向攻击与 HP 递减；
- 关键子任务正确性；
- 具体地图可达性；
- 带合理公平性前提的完备性。

## 6. Task5 策略证明

### 6.1 Task5 的完成条件

Task5 没有 `complete_task` 出口。Python 引擎要求世界中至少存在一个宝箱，
且所有模板房间的宝箱全部可见并打开后完成世界。

环境层新增：

```lean
allVisibleChestsOpened
allWorldChestsOpened
Step.completeAllChests
```

这保证 Task5 最终证明使用真实完成规则，而不是错误套用 Task1/2 的出口完成。

### 6.2 多房间记忆

`Task5Memory` 保存：

- `visitedRooms`
- `knownEdges`
- `exploredEdges`
- `staticBlockers`
- `openedChests`
- `pressedButtons`
- `learnedBlockedMoves`

相关更新函数全部采用只增不减的方式。

证明包括：

- 新房间一定进入 visited；
- 旧房间不会因访问新房间而丢失；
- explored edge 必然同时是 known edge；
- 旧静态阻挡不会被遗忘；
- 已开宝箱记忆不会倒退。

### 6.3 感知与记忆正确性契约

定义：

```lean
Task5MemorySound world memory
```

它要求：

- 记忆中的静态阻挡在对应真实房间中确实阻挡；
- 记忆中的已开宝箱确实存在且 `opened = true`；
- 记忆中的已按按钮在对应房间中确实 pressed。

关键定理：

- `sound_memory_static_blocker_is_real`
- `remember_real_static_blocker_preserves_soundness`

这明确了 CNN 和 Lean 之间的接口：CNN 不由 Lean 证明，但只有满足该契约的
符号记忆才进入后续安全证明。

### 6.4 换房确认

换房不读取隐藏 room ID，而使用：

- 上一帧玩家位于待推进方向的边界；
- 最后动作沿该出口方向；
- 当前玩家出现在房间严格内部的 spawn 附近；
- 玩家位置确实发生变化。

定义：

```lean
RoomTransitionEvidence
```

证明：

- 确认的换房一定从对应边界开始；
- 确认的换房一定结束在房间内部；
- 玩家 tile 未变化时不可能确认换房。

这对应 Python `_detect_room_transition`。

### 6.5 无硬编码高层目标选择

高层目标包括：

```text
slayMonster
openChest
pressButton
goExit
backtrack
wait
```

`Task5Candidates` 中的候选全部来自：

- 当前视觉可达宝箱；
- 当前视觉怪物；
- 当前视觉按钮；
- 当前视觉出口；
- 历史观测出口；
- DFS 探索栈。

`chooseTask5Goal` 的优先级为：

```text
阻挡宝箱的怪物
→ 可达宝箱
→ 使宝箱不可达的怪物
→ 未按按钮
→ 有钥匙时看到的锁门
→ 未探索普通出口
→ 回退出口
→ WAIT
```

证明：

- `chosen_goal_stays_in_current_room`
- `chosen_chest_came_from_visual_candidates`
- `chosen_button_came_from_visual_candidates`
- `chosen_exit_came_from_observed_or_backtrack_candidates`
- `reachable_chest_has_priority_over_button`

因此选择器不能凭空产生一个宝箱坐标或出口方向。

此外，`Task5GoalAchieved` 为清怪、开箱、按钮和换房目标定义可检查后置条件；
`Task5ControllerRefinesSelection` 要求局部控制器严格执行
`chooseTask5Goal` 的结果，并提供完整 `EngineExec` 达成证书。相关定理证明：

- 开箱控制器打开的位置来自视觉候选；
- 出口控制器使用的方向和位置来自已观测出口；
- 非 WAIT 目标必然给出完整 tick 执行及目标达成证明。

公开代码中的方向顺序只是候选出口之间的 tie-breaker，不表达“钥匙一定在
south”之类的隐藏知识。

### 6.6 四种移动模式

定义：

```lean
MoveMode.normal
MoveMode.combat
MoveMode.exit
MoveMode.rush
```

- `normal`：必须是 safe tile，并避开怪物邻域；
- `combat`：允许到安全攻击位；
- `exit`：允许靠近出口，但仍须物理可通行；
- `rush`：后期允许缩短绕路，但进入危险邻域必须已激活盾牌。

所有模式还必须避开 `learnedBlockedMoves`，即通过连续视觉反馈学到的像素碰撞
阻挡边。

关键证明：

- `task5_allowed_move_is_enterable`
- `task5_normal_move_avoids_monster_neighborhood`
- `task5_allowed_move_avoids_learned_blocked_edge`
- `task5Shield_blocks_disallowed_move`
- `task5Shield_allowed_move_is_enterable`
- `task5_unsafe_queued_move_is_interrupted`
- `task5_interrupted_move_is_masked`
- `verified_shield_flag_means_owned`
- `verified_rush_into_danger_has_active_shield`

最终 `task5Shield` 使用 `Task5VerifiedMoveAllowed`：`hasShield` 必须与真实
inventory 一致，rush 进入危险邻域时还必须满足 `player.shielding = true`。

### 6.7 局部子任务正确性

分别证明：

- `pressing_existing_button_records_pressed`
- `rotating_existing_bridge_changes_orientation`
- `opening_visible_chest_records_open`
- `task5_shielded_contact_preserves_health`
- `task5_successful_exit_changes_room`

它们说明：

- 踩按钮后能在房间状态中找到 pressed 按钮；
- switch 作用于已有桥时，目标房间中的桥朝向确实切换；
- 开箱后能在房间列表中找到 opened 宝箱；
- 盾牌格挡不降低 HP；
- 成功出口准确切换到目标房间和 spawn；锁门首次通过会持久写入 opened 并
  消耗配置钥匙，之后再次通过不会重复消费。

### 6.8 房间图探索完备性

定义：

```lean
RoomReachable graph start room
FairExplorationClosed graph visited
```

`RoomReachable` 是房间图的归纳可达关系：

- start 可达；
- 若 source 可达且存在 source → target 边，则 target 可达。

公平探索闭包表示：

```text
若 source 已访问
且 graph 中存在 source → target
则 target 最终也会被访问
```

主要定理：

- `fair_exploration_visits_every_reachable_room`
- `fair_exploration_visits_all_required_rooms`

证明使用 `RoomReachable` 的结构归纳，而不是枚举某条固定房间路线。

### 6.9 宝箱调度与世界完成

定义：

```lean
AllObjectiveChestsVisible
ChestSchedulerComplete
```

前者表示所有目标宝箱最终均可观察，后者表示公平目标调度器最终打开每一个
房间中的每一个宝箱。

证明：

- `fair_exploration_and_local_service_complete_scheduler`
- `fair_exploration_and_local_reveal_make_chests_visible`
- `visibility_and_fair_scheduler_open_all_chests`
- `all_chests_objective_completes_world`
- `task5_completable_after_all_chests_opened`
- `task5_policy_complete_under_assumptions`
- `task5_verified_policy_is_safe_and_complete`

完整逻辑为：

```text
有限非空房间集
∧ 房间图连通
∧ 公平探索访问所有必要房间
∧ 符号记忆正确
∧ 已访问房间的局部控制器完成开箱和揭示
∧ 玩家存活
→ 公平探索推出所有房间已访问
→ 推出所有宝箱可见且 scheduler 完成
→ allWorldChestsOpened
→ Step.completeAllChests
→ WorldCompleted ∧ alive ∧ ValidState
```

### 6.11 Task5 为什么是条件完备性

Task5 中存在以下不可无条件保证的外部行为：

- CNN 可能误识别；
- 怪物可能动态封锁路径；
- 玩家可能因时间或怪物伤害死亡；
- 某出口可能暂时无法通过；
- 某宝箱可能因动态危险暂时不可达。

旧的组合定理保留 `Task5Assumptions` 便于分层复用；加强版主定理使用
`Task5VerifiedAssumptions`，明确列出：

- 符号记忆正确；
- 执行轨迹满足完整 `EngineExec` tick 调度；
- 初态和终态安全条件；
- 房间图中的目标房间均可达；
- 探索结果对房间图公平闭合；
- 已访问房间由局部控制器完成宝箱揭示与开启；
- 玩家最终存活。

`ChestSchedulerComplete` 和 `AllObjectiveChestsVisible` 不再作为加强版主定理
的直接假设，而由公平探索和局部服务定理推导。

**对应踩分点：**

- 多房间状态与探索策略形式化；
- 高层目标选择器；
- BFS 和 action mask；
- 历史记忆与视觉契约；
- 动态队列安全；
- 宝箱、按钮、桥、怪物、出口等关键机制；
- 房间图可达性；
- 公平探索完备性；
- 世界完成正确性；
- 无固定目标坐标的通用策略。
